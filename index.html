What is a "static" web app?

A static app is just files (HTML, CSS, JS) served to the browser. No server running your code. The browser does all the work.

Static

  • Just files
  • Free to host
  • Fast
  • Talks to external APIs for data

Dynamic

  • Needs a server running 24/7
  • Usually costs money
  • More moving parts
  • More power, more complexity

Today: static frontend + Google Apps Script as our "backend in the cloud."

Google Sheets as your database

For most small apps, a spreadsheet is more than enough. You already know how to use it. It's free. It's shareable.

  • Rows = records
  • Columns = fields
  • First row = column names
  • Visible — you can edit/inspect data anytime

Designing your sheet

Example: a "users" sheet for a sign-up form.

idnameemailcreated_at

2Benben@example.com2026-05-27

Tip: name your sheet tab clearly (e.g. "users"). We'll reference that name in code.

When Sheets is NOT enough

It's a great start, but know the limits.

  • Hundreds of writes per second? No.
  • Millions of rows? No.
  • Sensitive personal data? Be careful.
  • Complex relations? Possible but messy.

For 99% of side projects and class projects, Sheets is perfect.

The two functions you need

Apps Script Web Apps run two special functions when a URL is called:

  • doGet(e) — runs when someone reads data (GET request)
  • doPost(e) — runs when someone writes data (POST request)

Each receives an e (event) object with parameters and body.

Reading from the sheet

A minimal doGet that returns all rows as JSON.

function doGet(e) {
  const sheet = SpreadsheetApp
    .getActiveSpreadsheet()
    .getSheetByName("users");

  const data = sheet.getDataRange().getValues();
  const headers = data.shift();

  const rows = data.map(row => {
    const obj = {};
    headers.forEach((h, i) => obj[h] = row[i]);
    return obj;
  });

  return ContentService
    .createTextOutput(JSON.stringify(rows))
    .setMimeType(ContentService.MimeType.JSON);
}

Writing to the sheet

A minimal doPost that appends a new row.

function doPost(e) {
  const body = JSON.parse(e.postData.contents);

  const sheet = SpreadsheetApp
    .getActiveSpreadsheet()
    .getSheetByName("users");

  sheet.appendRow([
    Date.now(),
    body.name,
    body.email,
    new Date()
  ]);

  return ContentService
    .createTextOutput(JSON.stringify({ ok: true }))
    .setMimeType(ContentService.MimeType.JSON);
}

Deploy as a Web App

In Apps Script editor:

  1. Click Deploy → New deployment
  2. Type: Web app
  3. Execute as: Me
  4. Who has access: Anyone
  5. Click Deploy — copy the URL

That URL is your API endpoint. Treat it as semi-public.

Common gotchas

  • CORS: Apps Script Web Apps return JSON without CORS headers — use simple GET/POST without custom headers and you're fine. Avoid Content-Type: application/json on POST; send as plain text or form data.
  • Redeploy: after editing code, you must create a new version or manage deployments — old URL keeps running old code otherwise.
  • Permissions: first run asks you to authorize. That's normal.
  • Quotas: free tier handles thousands of requests/day — plenty for class projects.

Calling the API from your page

Browsers have a built-in function called fetch() for talking to APIs.

const url = "https://script.google.com/macros/s/AKfycb.../exec";

fetch(url)
  .then(res => res.json())
  .then(data => {
    console.log(data);
  });

Open the browser console (F12) to see the result.

Sending data (POST)

fetch(url, {
  method: "POST",
  body: JSON.stringify({
    name: "Aisha",
    email: "aisha@example.com"
  })
})
  .then(res => res.json())
  .then(data => console.log("Saved:", data));

No custom headers = no CORS preflight = no headaches.

What is GitHub?

GitHub is where you store your code online. Think of it as Google Drive for code — with version history.

  • Free for public projects
  • Tracks every change you make
  • Required for free hosting (next section)

Creating your repo

  1. Sign in at github.com
  2. Click New repository
  3. Give it a name (e.g. my-first-app)
  4. Public · Add README · Create
  5. Upload your index.html (drag & drop works!)

You don't need to learn Git commands today — the web UI is enough.

Deploy with GitHub Pages

  1. In your repo: Settings → Pages
  2. Source: Deploy from branch
  3. Branch: main / root
  4. Save · wait ~1 minute
  5. Your site is at username.github.io/repo-name

Or: Cloudflare Pages

A nicer experience with a faster network and unlimited bandwidth on the free tier.

  1. Sign in at pages.cloudflare.com
  2. Create project · connect your GitHub repo
  3. Framework: None · Build command: (empty)
  4. Output directory: /
  5. Deploy · you get a .pages.dev URL

Every push to GitHub auto-deploys. Magic.
